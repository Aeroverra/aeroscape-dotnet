# Audit Report 3: Combat, Magic, Skills, Prayer

## Critical Bugs

- PvE special-attack multipliers do not match Java. The C# port reuses one shared `WeaponData.SpecialAttacks` table for both PvP and PvE ([AeroScape.Server.Core/Combat/WeaponData.cs](../AeroScape.Server.Core/Combat/WeaponData.cs), [AeroScape.Server.Core/Combat/PlayerVsNpcCombat.cs](../AeroScape.Server.Core/Combat/PlayerVsNpcCombat.cs)), but `PlayerNPCCombat.java` uses different NPC-only multipliers: AGS `1.25`, BGS `1.1`, SGS `1.1`, ZGS `1.1`, whip `0.9`, dragon scimitar `1.0` ([legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java)). C# currently applies PvP values such as AGS `1.6`, whip `1.3`, d scim `1.3`, which materially changes PvE DPS.

- Dragon claws are incomplete in both PvP and PvE. Java lands the later claw hits through `clawTimer` / `clawTimer2`, `UseClaws` / `UseClaws2`, and delayed follow-up methods in `Player.java` ([legacy-java/server508/src/main/java/DavidScape/players/Player.java:2916](../legacy-java/server508/src/main/java/DavidScape/players/Player.java#L2916), [legacy-java/server508/src/main/java/DavidScape/players/Player.java:136](../legacy-java/server508/src/main/java/DavidScape/players/Player.java#L136)). C# explicitly leaves hits 3 and 4 as TODOs in both combat paths ([AeroScape.Server.Core/Combat/PlayerVsPlayerCombat.cs:247](../AeroScape.Server.Core/Combat/PlayerVsPlayerCombat.cs#L247), [AeroScape.Server.Core/Combat/PlayerVsNpcCombat.cs:107](../AeroScape.Server.Core/Combat/PlayerVsNpcCombat.cs#L107)).

- Vengeance is not ported in PvP. Java reflects `75%` of the incoming hit, force-chats `"Taste Vengeance!"`, and clears `vengOn` before the main hit resolves ([legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java:34](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java#L34)). C# leaves `ProcessVengeance` as a no-op stub ([AeroScape.Server.Core/Combat/PlayerVsPlayerCombat.cs:246](../AeroScape.Server.Core/Combat/PlayerVsPlayerCombat.cs#L246)).

- Protection prayers were rewritten as random blocking instead of the Java `Hitter` counter. Java uses deterministic counter state on the target (`Hitter > 0 => full block, else reseed to 2..6`) for PvP ranged/melee and NPC melee ([legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java:154](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java#L154), [legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java:331](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java#L331), [legacy-java/server508/src/main/java/DavidScape/npcs/combat/NPCPlayerCombat.java:59](../legacy-java/server508/src/main/java/DavidScape/npcs/combat/NPCPlayerCombat.java#L59)). C# replaced this with `Random(4) < 3` in both PvP and NPC combat ([AeroScape.Server.Core/Combat/PlayerVsPlayerCombat.cs:278](../AeroScape.Server.Core/Combat/PlayerVsPlayerCombat.cs#L278), [AeroScape.Server.Core/Combat/NpcVsPlayerCombat.cs:137](../AeroScape.Server.Core/Combat/NpcVsPlayerCombat.cs#L137)). That is not an exact port.

- `Player.Hitter` starts with the wrong default. Java initializes `Hitter = 5` ([legacy-java/server508/src/main/java/DavidScape/players/Player.java:324](../legacy-java/server508/src/main/java/DavidScape/players/Player.java#L324)); C# leaves it at the CLR default `0` ([AeroScape.Server.Core/Entities/Player.cs:168](../AeroScape.Server.Core/Entities/Player.cs#L168)). Even before the prayer-logic rewrite above, the initial protection-prayer behavior is wrong.

- PvP ranged damage is not an exact port. Java seeds `hitDamage` once from `maxMeleeHit(p)` at the top of `attackPlayer` and reuses it for the bow branch ([legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java:34](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java#L34), [legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java:163](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java#L163)). C# recalculates ranged hits from `MaxRangeHit` ([AeroScape.Server.Core/Combat/PlayerVsPlayerCombat.cs:201](../AeroScape.Server.Core/Combat/PlayerVsPlayerCombat.cs#L201)). That is a large behavioral divergence.

- PvE crystal-bow damage is not an exact port. Java’s crystal-bow NPC branch uses the `hitDamage` value seeded from melee max-hit ([legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java:44](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java#L44), [legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java:213](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java#L213)). C# uses ranged max-hit for the same branch ([AeroScape.Server.Core/Combat/PlayerVsNpcCombat.cs:183](../AeroScape.Server.Core/Combat/PlayerVsNpcCombat.cs#L183)).

- Superheat can pick the wrong bar. Java distinguishes `440` with no coal as iron, `440 + 2 coal` as steel, and `440 + 1 coal` as invalid ([legacy-java/server508/src/main/java/DavidScape/players/Magic.java:600](../legacy-java/server508/src/main/java/DavidScape/players/Magic.java#L600), [legacy-java/server508/src/main/java/DavidScape/players/Magic.java:616](../legacy-java/server508/src/main/java/DavidScape/players/Magic.java#L616), [legacy-java/server508/src/main/java/DavidScape/players/Magic.java:633](../legacy-java/server508/src/main/java/DavidScape/players/Magic.java#L633)). C# resolves a recipe by the first ore-id match, so `FindSmeltByOre(440)` always resolves to the iron-bar recipe before steel ([AeroScape.Server.Core/Skills/SmithingSkill.cs:133](../AeroScape.Server.Core/Skills/SmithingSkill.cs#L133), [AeroScape.Server.Core/Skills/SmithingSkill.cs:243](../AeroScape.Server.Core/Skills/SmithingSkill.cs#L243), [AeroScape.Server.Core/Skills/SmithingSkill.cs:311](../AeroScape.Server.Core/Skills/SmithingSkill.cs#L311)).

- Combat magic coverage is far below the Java implementation. Java `Magic.java` contains the full modern combat spellbook, the entire ancient spellbook, magic-on-player combat, teleports, utility spells, superheat, enchants, charge, and lunar vengeance ([legacy-java/server508/src/main/java/DavidScape/players/Magic.java](../legacy-java/server508/src/main/java/DavidScape/players/Magic.java)). C# currently implements only a small modern utility subset plus the 16 standard autocast NPC spells ([AeroScape.Server.Core/Services/MagicService.cs](../AeroScape.Server.Core/Services/MagicService.cs), [AeroScape.Server.Core/Combat/MagicSpellData.cs](../AeroScape.Server.Core/Combat/MagicSpellData.cs)). Most combat magic behavior from Java is absent.

## Logic Errors

- Zamorak godsword energy does not match Java. Both Java combat paths gate ZGS on `>= 75` energy but subtract only `50` when the hit fires ([legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java:243](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerCombat.java#L243), [legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java:298](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java#L298)). C# charges the full `75` via `WeaponData.SpecialAttacks` ([AeroScape.Server.Core/Combat/WeaponData.cs:69](../AeroScape.Server.Core/Combat/WeaponData.cs#L69)).

- Slayer XP for dragon tasks is wrong. Java task `0` awards `150 * slayerLevel` for npc types `941`, `55`, and `53`, and `500 * slayerLevel` only for `5363` ([legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java:165](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java#L165)). C# collapses the whole group and awards `250 * slayerLevel` for every non-`5363` dragon ([AeroScape.Server.Core/Combat/PlayerVsNpcCombat.cs:315](../AeroScape.Server.Core/Combat/PlayerVsNpcCombat.cs#L315)).

- Autocast no longer requires a staff. Java only autocasts NPC spells if `hasStaff()` is true ([legacy-java/server508/src/main/java/DavidScape/players/combat/MagicNPC.java:57](../legacy-java/server508/src/main/java/DavidScape/players/combat/MagicNPC.java#L57), [legacy-java/server508/src/main/java/DavidScape/players/combat/MagicNPC.java:518](../legacy-java/server508/src/main/java/DavidScape/players/combat/MagicNPC.java#L518)). C# has a `HasAutocastStaff` helper but never uses it in `ProcessMagicAttack` or `ProcessAttack` ([AeroScape.Server.Core/Combat/MagicNpcService.cs:11](../AeroScape.Server.Core/Combat/MagicNpcService.cs#L11), [AeroScape.Server.Core/Combat/PlayerVsNpcCombat.cs:65](../AeroScape.Server.Core/Combat/PlayerVsNpcCombat.cs#L65)).

- PvE specials are enabled for weapons Java never allowed against NPCs. `PlayerNPCCombat.java` has no anger-weapon handling and no saradomin-sword spec branch, but C# exposes both via the shared special table ([legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java](../legacy-java/server508/src/main/java/DavidScape/players/combat/PlayerNPCCombat.java), [AeroScape.Server.Core/Combat/WeaponData.cs:83](../AeroScape.Server.Core/Combat/WeaponData.cs#L83)).

- Teleports do not follow Java’s delayed teleport flow. Java uses `teleportTo(...)` / `ancientTeleTo(...)`, then resolves movement and finish gfx/anim later when `teleDelay == 0` ([legacy-java/server508/src/main/java/DavidScape/players/Magic.java:77](../legacy-java/server508/src/main/java/DavidScape/players/Magic.java#L77), [legacy-java/server508/src/main/java/DavidScape/players/Player.java:4382](../legacy-java/server508/src/main/java/DavidScape/players/Player.java#L4382)). C# teleports by directly overwriting `AbsX` / `AbsY` in `CastTeleport` ([AeroScape.Server.Core/Services/MagicService.cs:163](../AeroScape.Server.Core/Services/MagicService.cs#L163)).

- Fishing spot definitions were moved into C# and mostly match, but the C# tool/message model diverges from the actual Java handlers. Java option-1 for NPC `316` and `313` checks item `305` and even messages `"big fishing net"` ([legacy-java/server508/src/main/java/DavidScape/io/packets/NPCOption1.java:694](../legacy-java/server508/src/main/java/DavidScape/io/packets/NPCOption1.java#L694)); C# relabels `316` option 1 as `"small fishing net"` and hardcodes spot metadata in a new table ([AeroScape.Server.Core/Skills/FishingSkill.cs:51](../AeroScape.Server.Core/Skills/FishingSkill.cs#L51)). The catch values align, but the source-of-truth is no longer the legacy flow.

- `PrayerService.Reset` clears all 27 prayers, while Java’s `resetPrayers` only iterates `0..20` and leaves the rest to other code paths ([legacy-java/server508/src/main/java/DavidScape/io/packets/Prayer.java:251](../legacy-java/server508/src/main/java/DavidScape/io/packets/Prayer.java#L251), [AeroScape.Server.Core/Services/PrayerService.cs:84](../AeroScape.Server.Core/Services/PrayerService.cs#L84)). If the extra prayers are intended to stay legacy-accurate, this is another mismatch.

## Missing Implementations

- Ancient magic is missing. Java ships full ancient level tables, XP tables, max-hit tables, rune requirements, and combat-on-player handling ([legacy-java/server508/src/main/java/DavidScape/players/Magic.java:2311](../legacy-java/server508/src/main/java/DavidScape/players/Magic.java#L2311)). No equivalent exists in C# magic services or combat classes.

- Modern player-vs-player magic is missing. Java handles offensive modern spells on players, including spell-specific rune checks and XP formulas (`hit * 4 + baseXp`) inside `Magic.java` ([legacy-java/server508/src/main/java/DavidScape/players/Magic.java:1727](../legacy-java/server508/src/main/java/DavidScape/players/Magic.java#L1727)). C# has no comparable player-combat magic pipeline.

- Autocast state is incomplete in `Player.cs`. Java `Player` carries `magicNPC`, `modernSpell`, `usingAutoCast`, `castAuto`, `modernMagicDelay`, and related control fields ([legacy-java/server508/src/main/java/DavidScape/players/Player.java:116](../legacy-java/server508/src/main/java/DavidScape/players/Player.java#L116), [legacy-java/server508/src/main/java/DavidScape/players/Player.java:183](../legacy-java/server508/src/main/java/DavidScape/players/Player.java#L183)). C# only keeps `AutoCasting`, `AutoCastSpellId`, and `MagicCanCast` ([AeroScape.Server.Core/Entities/Player.cs:147](../AeroScape.Server.Core/Entities/Player.cs#L147)).

- Smithing products are incomplete. Java smithing exposes extra products such as `Items1` (wire/split/sheet), `Lantarn` (oil lantern frame / bullseye lantern), and `GrapleHook` (grapple tip) in the smithing interface ([legacy-java/server508/src/main/java/DavidScape/Skills/Smithing.java:32](../legacy-java/server508/src/main/java/DavidScape/Skills/Smithing.java#L32), [legacy-java/server508/src/main/java/DavidScape/Skills/Smithing.java:1406](../legacy-java/server508/src/main/java/DavidScape/Skills/Smithing.java#L1406), [legacy-java/server508/src/main/java/DavidScape/Skills/Smithing.java:1550](../legacy-java/server508/src/main/java/DavidScape/Skills/Smithing.java#L1550), [legacy-java/server508/src/main/java/DavidScape/Skills/Smithing.java:1568](../legacy-java/server508/src/main/java/DavidScape/Skills/Smithing.java#L1568)). C# `Products` omits all of them ([AeroScape.Server.Core/Skills/SmithingSkill.cs:88](../AeroScape.Server.Core/Skills/SmithingSkill.cs#L88)).

- The smithing UI is mostly TODO in C#. Java populates icons, bar counts, and per-tier level labels through `smithtab`, `Iof`, `amoutofbars`, and `LvlReq` ([legacy-java/server508/src/main/java/DavidScape/Skills/Smithing.java](../legacy-java/server508/src/main/java/DavidScape/Skills/Smithing.java)). C# `OpenSmithingInterface` only stores `CurrentMetalType` and leaves the rest as comments ([AeroScape.Server.Core/Skills/SmithingSkill.cs:155](../AeroScape.Server.Core/Skills/SmithingSkill.cs#L155)).

- `Player.cs` does not match `Player.java` field coverage. A quick field inventory found `513` public Java fields vs `275` public C# fields, with `272` Java fields lacking even a normalized counterpart. Scope-relevant missing examples include `actionButtonTimer`, `objectTimer`, `magicNPC`, `modernDamageDelay`, `modernMagicDelay`, `modernSpell`, `usingAutoCast`, `castAuto`, `secHit2`, `thirdHit2`, `fourHit2`, `clawTimer2`, `UseClaws2`, `HeadTimer`, `TeleBackTimer`, and `SlayerCaveTimer` ([legacy-java/server508/src/main/java/DavidScape/players/Player.java](../legacy-java/server508/src/main/java/DavidScape/players/Player.java), [AeroScape.Server.Core/Entities/Player.cs](../AeroScape.Server.Core/Entities/Player.cs)).

- The secondary dragon-claw NPC fields are missing from `Player.cs`. Java uses `secHit2`, `thirdHit2`, `fourHit2`, `clawTimer2`, and `UseClaws2` for PvE claw follow-up hits ([legacy-java/server508/src/main/java/DavidScape/players/Player.java:123](../legacy-java/server508/src/main/java/DavidScape/players/Player.java#L123), [legacy-java/server508/src/main/java/DavidScape/players/Player.java:141](../legacy-java/server508/src/main/java/DavidScape/players/Player.java#L141)). C# has no equivalent PvE state ([AeroScape.Server.Core/Entities/Player.cs:161](../AeroScape.Server.Core/Entities/Player.cs#L161)).

## Minor Issues

- Woodcutting and mining data tables are largely accurate, but the C# comments are misleading around “speed values”. Java defines axe/pickaxe speed constants, then immediately collapses `time` back to `4` for woodcutting and almost always `4` for mining (`1` is overwritten for rune essence too) ([legacy-java/server508/src/main/java/DavidScape/Skills/Woodcutting.java:316](../legacy-java/server508/src/main/java/DavidScape/Skills/Woodcutting.java#L316), [legacy-java/server508/src/main/java/DavidScape/Skills/Mining.java:367](../legacy-java/server508/src/main/java/DavidScape/Skills/Mining.java#L367)). C# hardcodes `4`, which matches effective behavior, but does not preserve the original intermediate speed state.

- Prayer toggle logic does not mirror Java’s interface config updates. Java calls `frames.setConfig` for the toggled prayer and each forced-off prayer ([legacy-java/server508/src/main/java/DavidScape/io/packets/Prayer.java:14](../legacy-java/server508/src/main/java/DavidScape/io/packets/Prayer.java#L14), [legacy-java/server508/src/main/java/DavidScape/io/packets/Prayer.java:239](../legacy-java/server508/src/main/java/DavidScape/io/packets/Prayer.java#L239)). C# only flips `PrayOn` and updates state flags ([AeroScape.Server.Core/Services/PrayerService.cs:57](../AeroScape.Server.Core/Services/PrayerService.cs#L57)).

- Java emits user-facing feedback that is missing in many C# skill and magic paths. Examples include woodcutting/mining inventory-full messages, fishing bait/tool failures, smithing bar exhaustion, and prayer level failures ([legacy-java/server508/src/main/java/DavidScape/Skills/Woodcutting.java:66](../legacy-java/server508/src/main/java/DavidScape/Skills/Woodcutting.java#L66), [legacy-java/server508/src/main/java/DavidScape/Skills/Mining.java:78](../legacy-java/server508/src/main/java/DavidScape/Skills/Mining.java#L78), [legacy-java/server508/src/main/java/DavidScape/io/packets/Prayer.java:30](../legacy-java/server508/src/main/java/DavidScape/io/packets/Prayer.java#L30)). Most corresponding C# paths are still TODO comments.
